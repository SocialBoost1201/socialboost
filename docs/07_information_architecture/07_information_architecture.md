# SocialBoost 公式Webサイト 情報設計書

---

## 文書目的

本書は、SocialBoost 公式Webサイトにおいて、どの情報を、誰に、どの順番で、どの粒度で、どのページに配置するかを定義するための文書である。
本書の目的は、見た目を作る前に情報の構造を整理し、ユーザーが迷わず理解し、比較検討し、相談行動まで自然に進める情報導線を設計することにある。
本書は、要件定義書、基本設計書、詳細設計書を受けて、後続のサイトマップ設計、ページ設計、SEO設計、UI設計、コピー設計の基準とする。

## 前提

本フェーズでは実装を行わない。
本書は情報設計文書であり、コード生成、UI構築、ページ実装、依存追加、ルーティング実装、CMS実装、データ投入、デプロイは行わない。
ユーザー本人が明示的に実装許可を出すまで、実装フェーズへ進まない。

---

## 情報設計の基本思想

SocialBoost は、単なる制作会社としてではなく、事業課題を整理し、必要に応じてWeb制作、LP制作、Webシステム開発、アプリ開発、AI導入支援、改善運用まで一気通貫で支援できる **事業成長パートナー** として見せる必要がある。
したがって、本サイトの情報設計は、制作物を見せることだけを目的にせず、以下を同時に満たす構造でなければならない。

1. 何をしている事業かが一瞬で分かること
2. 何が頼めるかが整理されていること
3. どの程度信頼できるかを実績で補強できること
4. 相談しやすいと感じられること
5. 料金感や依頼範囲の不安を下げられること
6. 将来的な実績追加、サービス追加、資料追加に耐えられること

---

## 情報設計の目的

1. 理解コストを下げる
2. 比較検討コストを下げる
3. 問い合わせ前の不安を下げる
4. 営業資料としても使える構造を持つ
5. SEOとGEOの両方に耐える構造を持つ
6. 実績一覧から案件詳細、画像、PDFへ自然につなげる
7. 個人事業主段階でも不自然なく信頼形成できる構造を持つ

## 対象ユーザーごとの情報ニーズ整理

### 一次ターゲット

中小企業経営者 / 個人事業主 / 店舗事業者 / 新規事業立ち上げ検討者 / 法人担当者

#### 一次ターゲットが知りたいこと

- この人は何ができるのか
- ただの制作屋ではないのか
- どこまで対応してくれるのか
- 過去にどんなものを作ってきたのか
- どのくらいの価格帯なのか
- どうやって相談すればいいのか

### 二次ターゲット

既存サイト改善を検討している事業者 / システムやアプリを作りたい事業者 / AI活用を検討している事業者

#### 二次ターゲットが知りたいこと

- 自分の課題がこのサイトで扱われているか
- システムやアプリも本当にできるのか
- MVPや小規模スタートにも対応できるのか
- 制作だけで終わらず改善まで見てもらえるのか
- 相談前にどの程度判断材料が得られるのか

## 情報優先順位

サイト全体で優先して伝えるべき情報の順序は以下とする。

| 優先度 | 情報 |
|:---|:---|
| 第1優先 | 何をしている事業か |
| 第2優先 | 何ができるか |
| 第3優先 | どんな実績があるか |
| 第4優先 | なぜ選ばれるか |
| 第5優先 | どのくらいの価格感か |
| 第6優先 | どう相談するか |
| 第7優先 | 補助情報（法務情報、将来拡張情報） |

この優先順位を崩して、低優先情報を主要導線に混ぜすぎないことを原則とする。

## 情報レイヤー設計

本サイトの情報は、以下の3レイヤーに分けて整理する。

| レイヤー | 名称 | 内容 |
|:---|:---|:---|
| レイヤー1 | 理解導入情報 | 何者か、何ができるか、どのような課題に対応できるか |
| レイヤー2 | 信頼形成情報 | 実績、選ばれる理由、進め方、対応範囲、料金の考え方 |
| レイヤー3 | 行動誘導情報 | お問い合わせ、無料相談、実績詳細閲覧、料金確認、サービス詳細確認 |

全ページは、この3レイヤーのどれに属するかを意識して設計する。

## トップページの情報順序設計

トップページは最重要ページであり、情報順序を誤ると離脱率が高まるため、以下の順で配置する。

| No. | セクション | 情報の目的 |
|:---|:---|:---|
| 1 | ファーストビュー | 何をしている事業か、一気通貫の支援範囲、主要CTA |
| 2 | 課題訴求 | 誰のどんな悩みに応えるか |
| 3 | サービス概要 | 何ができるか |
| 4 | 実績プレビュー | どのようなものを作ってきたか |
| 5 | 選ばれる理由 | なぜ依頼する価値があるか |
| 6 | 料金案内 | どのくらいの価格感か |
| 7 | 進行フロー | どう進むのか |
| 8 | 不安解消 | 相談前の不安をなくす |
| 9 | お問い合わせCTA | 最終行動導線 |

この順序は、**理解 → 信頼 → 安心 → 行動** の流れに基づいている。

## サービス情報の情報粒度設計

### 一覧ページで持つ粒度

- サービス名
- 1文説明
- 対象課題
- 詳細ページ導線

### 詳細ページで持つ粒度

- 対象顧客
- よくある課題
- 提供内容
- 対応範囲
- 進行の流れ
- 関連実績
- 料金導線
- 問い合わせ導線

### 一覧で説明しすぎない理由

サービス一覧で長文説明を行うと理解速度が落ちるため、一覧は比較可能な粒度に揃える。
詳細理解は個別ページに委譲する。

## 実績情報の情報粒度設計

### 実績一覧で持つ粒度

- サムネイル
- 案件名
- カテゴリ
- 業種
- 短い説明

### 実績詳細で持つ粒度

- 案件名
- カテゴリ
- 業種
- 概要
- 課題
- 対応範囲
- 実施内容
- 成果または狙い
- 画像一覧
- PDF一覧
- 関連実績
- 問い合わせ導線

### 画像とPDFの情報分離方針

- 画像は視覚的な制作物理解を目的とする
- PDFは提案資料、設計資料、営業資料などの補足理解を目的とする
- この2つは目的が異なるため、同一一覧や同一セクションで混在させない

### 関連実績の情報役割

- 関連実績は回遊導線として扱い、現在の案件理解を邪魔しない情報量に抑える
- 主役は現在閲覧中の案件であり、関連実績は補助導線とする

## 価格情報の情報粒度設計

### トップページで持つ粒度

- 参考価格があること
- 料金の考え方を見られること
- 料金ページへの導線

### 料金ページで持つ粒度

- サービス別参考レンジ
- 変動要因
- 見積もり前提
- 小規模スタート案内
- 問い合わせ導線

### 載せない方がよい情報

- 極端に細かいケース別個別価格
- 見積もり前の断定価格
- 説明のない最安値訴求

**理由**: ブランド価値毀損とミスリードを避けるため。

## 選ばれる理由の情報設計

このページまたはセクションでは、理念だけを語るのではなく、比較判断に使える具体情報を持たせる。

### 優先して載せる情報

- 事業理解から入ること
- 制作だけで終わらないこと
- システム、アプリ、AIまで視野に入れられること
- 必要なものを整理して提案すること
- 小さく始められること
- 相談しやすいこと

### 避ける情報

- 抽象的な美辞麗句のみ
- 長すぎる自己紹介
- 感情的すぎるストーリー過多

## 問い合わせ情報の情報設計

お問い合わせページでは、入力項目より先に安心感を与える情報を置く。
何を頼めばいいか分からなくても相談できることを明記する。

### 情報順序

1. ページタイトル
2. 歓迎メッセージ
3. 簡潔な補足
4. フォーム
5. 送信後の案内想定

### 入力項目優先順位

1. 氏名
2. メールアドレス
3. 相談内容
4. 相談種別
5. 会社名または屋号
6. 電話番号
7. 希望連絡方法

### 設計方針

必須項目は必要最小限にする前提を持つ。
ただし詳細な必須定義はフォーム設計書で確定する。

## 導線設計

### 主導線

- トップページ → お問い合わせ
- サービス詳細 → お問い合わせ
- 料金ページ → お問い合わせ
- 実績詳細 → お問い合わせ
- 選ばれる理由 → お問い合わせ

### 補助導線

- トップ → サービス
- トップ → 料金
- トップ → 実績
- サービス → 関連実績
- 実績 → 関連実績
- 料金 → サービス
- 料金 → 問い合わせ

### 導線設計の原則

- 情報を読ませてから行動させる
- ただし押し売り感を出さず、いつでも相談できる逃げ道を残す
- 各主要ページに1つ以上の問い合わせ導線を持たせる
- 1ページに導線を増やしすぎて優先順位を曖昧にしない

## SEOを考慮した情報設計

- ページ役割を分ける
- サービス一覧とサービス詳細を分ける
- 実績一覧と実績詳細を分ける
- 価格情報は料金ページに集約する
- 選ばれる理由は信頼形成情報として独立価値を持たせる

### 見出し構造の前提

- ページごとに主見出しは1つ
- セクション見出しで意味を分ける
- 見出しと本文の関係が一目で分かるようにする

### 情報重複ルール

- 同じ説明を全ページにコピペしない
- トップは要約、詳細は具体、一覧は比較の役割を守る

## アセット情報設計

| 分類 | 対象 | 管理方針 |
|:---|:---|:---|
| favicon系 | ブラウザ認識用アイコン | `public` 直下管理前提 |

- faviconはサイト基盤情報

- として役割を分ける

---

## 情報設計上の制約

- 会社概要を主要導線に置かない
- 画像とPDFを混在表示しない
- トップページで長文説明をしすぎない
- 一覧ページで詳細説明をしすぎない
- 価格で安売り感を出さない
- アニメーション前提で情報構造を複雑化しない

## ページ間の情報関係図

```
トップページ ─┬─→ サービス概要 ─→ サービス詳細 ─┬─→ 関連実績
              ├─→ 料金ページ ──────────────────┤
              ├─→ 実績一覧 ──→ 実績詳細 ─┬─→ 画像一覧
              │                          ├─→ PDF一覧
              │                          ├─→ 関連実績
              │                          └─→ お問い合わせ
              ├─→ 選ばれる理由 ────────────┤
              └─→ お問い合わせ ◀────────── 全主要ページ
```

この構造により、ユーザーはどの入口から入っても相談へ到達できる。
