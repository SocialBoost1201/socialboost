# SocialBoost 公式Webサイト GEO設計書

---

## 文書目的

本書は、SocialBoost 公式Webサイトにおいて、生成AI検索、AIアシスタント、要約エンジン、対話型検索体験に参照されやすい情報構造を整備するための設計方針を定義することを目的とする。
本書でいう GEO は、生成AI最適化の観点から、ページ構造、記述粒度、情報整理、引用されやすさ、比較されやすさ、要約されやすさ、誤解されにくさを高める設計を指す。
本書は、要件定義書、基本設計書、詳細設計書、情報設計書、SEO設計書を補完し、後続のコピー設計、構造化データ設計、FAQ設計、実装設計、運用設計の基準とする。

## GEO設計の基本思想

SocialBoost のサイトは、単に検索エンジンで見つかるだけでは不十分である。
今後は、AIアシスタントや生成AI検索において以下が短く正確に要約され、比較対象として扱われやすい構造を持つ必要がある。

- 何をしている事業か
- 何が依頼できるか
- どこまで対応できるか
- どのような実績があるか
- どのような料金思想か
- どのように相談できるか

長い抽象表現よりも、**要点が抜き出しやすく、誤解されにくい構造** を優先する。

---

## GEO設計の目的

1. 生成AIが SocialBoost の事業内容を誤解しにくいこと
2. 各ページが何のページかを短時間で要約できること
3. サービス、実績、料金、相談方法が構造的に把握できること
4. AIによる比較、要約、引用に耐える粒度で情報整理されていること
5. 画像やPDFに依存せず、本文だけでも十分に内容理解できること
6. FAQや要点整理により、AIに拾われやすい情報単位を持つこと
7. SEOと矛盾せず、むしろ補完し合う構造を持つこと

---

## GEO設計の基本方針

1. 各ページの冒頭で、そのページが何を扱うかを明確にする
2. 各ページに、短く要約しやすい説明文を持たせる
3. サービスページでは、誰向けか・何ができるか・どこまで対応できるかを分けて書く
4. 実績ページでは、案件概要・課題・対応範囲・実施内容・成果または狙いを明確に分ける
5. 料金ページでは、価格の考え方と参考レンジの考え方を整理する
6. お問い合わせページでは、相談可能範囲・要件未整理でも相談可能であること・送信後の流れを明示する
7. 画像やPDFに閉じた重要情報を本文に再表現する
8. 一覧ページは一覧として、詳細ページは詳細として要約可能な役割分担を持たせる
9. 抽象的な美辞麗句だけで構成しない
10. 箇条化しやすい情報は箇条的に整理可能な構造を持つ

| カテゴリ | 内容 |
|:---|:---|
| 事業概要 | SocialBoost は何をしている事業か |

## ページ別GEO設計方針

### トップページ

#### AIに伝わるべき内容

- SocialBoost は何をしているか
- どのような支援ができるか
- 制作だけでなくシステム、アプリ、AIまで対応すること
- 実績、料金、相談導線があること

#### 必要な要件

- 冒頭で事業内容を簡潔に表現する
- 主要サービスを短く列挙できる構造を持つ
- 選ばれる理由を抽象論で終わらせない
- 問い合わせ導線が明確である

---

### サービス一覧ページ

#### AIに伝わるべき内容

- どのようなサービス群があるか
- それぞれ何に関する支援か

#### 必要な要件

- サービス名だけでなく一文説明を持つ
- 一覧ページ内で比較しやすい粒度を揃える
- 詳細ページに進むとより具体的な内容が確認できることが分かる構造

---

### サービス詳細ページ

#### AIに伝わるべき内容

- そのサービスは誰向けか
- どんな課題に対応するか
- 何を提供するか
- どこまで対応するか
- どのように進むか
- 関連実績があるか

#### 必要な要件

- ページ冒頭に一文要約を持つ
- 対象顧客、よくある課題、提供内容、対応範囲を分けて書く
- 進行フロー、関連実績、料金導線、問い合わせ導線を持つ

---

### 料金ページ

#### AIに伝わるべき内容

- 参考レンジで案内していること
- 価格は一律ではないこと
- 変動要因があること
- 小規模スタートが可能であること
- 相談ベースで見積もり整理が可能であること

#### 必要な要件

- 価格の考え方を明示する
- 参考レンジを整理する
- 変動要因を箇条的に整理する
- 相談可能性を明示する

---

### 実績一覧ページ

#### AIに伝わるべき内容

- どのようなカテゴリの実績があるか
- どのような業種の案件があるか
- 各案件を詳細ページで深掘りできること

#### 必要な要件

- 各カードに案件名、カテゴリ、業種、短い説明を持つ
- 冒頭に、実績詳細で何が見られるかを簡潔に説明する

---

### 実績詳細ページ

#### AIに伝わるべき内容

- どんな案件だったか
- どんな課題があったか
- どこまで対応したか
- 何を実施したか
- どんな成果または狙いがあったか
- どんな画像やPDFがあるか

#### 必要な要件

- 案件概要、課題、対応範囲、実施内容、成果を明確に分離する
- 画像やPDFにのみ重要情報を置かない
- 本文を読めば案件の中身が分かること

---

### 選ばれる理由ページ

#### AIに伝わるべき内容

- 他社と何が違うか
- なぜ選ばれるか
- どのような進め方か

#### 必要な要件

- 理由を短い見出し単位で整理する
- 具体的に書く（事業理解、一気通貫、小規模スタート、相談しやすさ等）
- 抽象的な理念だけで終わらせない

---

### お問い合わせページ

#### AIに伝わるべき内容

- どんな相談が可能か
- 要件未整理でも相談可能か
- 何を入力すればよいか
- 送信後にどうなるか

#### 必要な要件

- 対象範囲を明示する
- 要件が固まっていなくても相談可能であることを明示する
- 送信後の流れを短く示す

## AIに引用されやすい記述粒度

### 理想的な記述単位

- 1文1意味
- 1セクション1テーマ
- 箇条として切り出しやすい文脈
- 見出しだけで意味が分かる構造

### 推奨記述パターン

| パターン | 例 |
|:---|:---|
| 定義型 | SocialBoost は、事業課題の整理から、Web制作、Webシステム開発、アプリ開発、AI導入支援、改善運用まで対応する事業成長支援サービスです。 |
| 対象型 | このサービスは、新規事業の立ち上げや、既存サイトの改善、業務効率化を検討している事業者向けです。 |
| 範囲型 | 対応範囲は、企画、情報設計、制作、開発、改善提案、運用支援です。 |
| 課題型 | よくある課題として、問い合わせにつながらない、情報整理が不十分、要件が固まっていない等があります。 |
| 成果型 | この案件では、情報整理、導線改善、画面設計を行い、問い合わせしやすい構成を目指しました。 |

### 避ける記述パターン

- **抽象的で意味が曖昧な表現のみ**（本質的な価値を最大化します / 未来を切り拓く支援をします 等）
- **画像依存の説明**（見れば分かる前提、資料を読めば分かる前提）
- **長すぎる複文**（AI要約時に意味が崩れやすい）

---

## ページ冒頭要約の設計方針

各主要ページは、冒頭で以下のいずれかを満たす構造を持つ。

1. このページで何が分かるか
2. このページは誰向けか
3. このページで扱うサービス、案件、料金の概要

### 目的

- AIがページ冒頭だけで概要を掴みやすくする
- ユーザーがページの価値を短時間で理解できる
- ページの役割が明確になる

---

## FAQ的情報の設計方針

GEOでは、よくある疑問とその回答は非常に重要である。
ただし全ページをFAQページ化しない。必要なページに、必要な疑問だけを短く整理する。

### FAQ的情報を置くべき候補ページ

トップページ / 料金ページ / お問い合わせページ / 選ばれる理由ページ / 将来のFAQページ

### FAQ的情報の対象例

- 要件が固まっていなくても相談できるか
- 小規模案件でも対応可能か
- 制作だけでなくシステムやアプリも相談可能か
- 価格はどのように決まるか
- 公開後の改善も依頼できるか

---

## サービス説明のGEO設計

各サービス詳細ページで最低限必要な要素：

サービスの一文定義 → 対象顧客 → 対応課題 → 提供内容 → 対応範囲 → 進め方 → 関連実績 → 問い合わせ導線

この構造により、AIは「このサービスは何か / 誰向けか / 何に強いか / どんな相談が可能か」を要約しやすくなる。

---

## 実績説明のGEO設計

各実績詳細ページで最低限必要な要素：

案件概要 → 課題 → 対応範囲 → 実施内容 → 成果または狙い → 画像があること → PDFがあること → 関連実績 → 問い合わせ導線

**特に重要な原則：**
- 画像一覧とPDF一覧の存在を本文でも説明すること
- 画像やPDFにのみ重要情報を置かないこと
- 本文を読めば案件の中身が分かること

---

## 料金説明のGEO設計

料金ページで最低限必要な要素：

参考レンジ → 価格の考え方 → 変動要因 → 小規模スタート可否 → 相談可能性

AIは価格ページを要約する際、「いくらか / どういう考え方か / どのような条件で変動するか / 小規模案件でも相談できるか」を取り出す可能性が高い。
価格だけではなく、**料金思想と相談導線をセットで記述する**。

---

## 問い合わせ説明のGEO設計

お問い合わせページで最低限必要な要素：

何の相談が可能か → 要件未整理でも相談可能か → 入力項目 → 送信後の流れ

「相談歓迎 / 対象範囲 / 入力のしやすさ / 送信後の安心感」を短く整理する。

---

## 比較されやすい情報の明文化

生成AIは、複数サービスや企業を比較して回答することがある。
SocialBoost が比較対象として扱われやすくするため、以下を明文化する。

- 何ができるか
- どこまで対応できるか
- 誰向けか
- 何が強みか
- どのような進め方か
- どのような料金思想か

この情報が曖昧だと、AIによる比較対象に入りにくくなる。

## GEO向け内部リンク設計

内部リンクの目的は、検索回遊だけでなく、AIがサイト内の関係性を読み取りやすくすることにもある。

```
トップページ       → サービス一覧 / 料金 / 実績一覧 / 選ばれる理由 / お問い合わせ
サービス詳細       → 関連実績 / 料金 / お問い合わせ
実績詳細           → 関連実績 / 関連サービス / お問い合わせ
料金ページ         → サービス詳細 / お問い合わせ
選ばれる理由       → 実績 / サービス / お問い合わせ
```

### 原則

- 文脈に合うこと
- リンク先の意味が明確であること
- 過剰に増やしすぎないこと

---

## GEO向けメディア設計方針

| メディア | 方針 |
|:---|:---|
| 画像 | 補助情報とする。altで最低限の意味を持たせる。画像だけで説明を完結させない |
| PDF | タイトルと概要説明を必ず持たせる。資料だけに重要情報を閉じ込めない |

---

## GEO向け構造の禁止事項

- 画像だけで価値を伝えようとすること
- PDFだけで案件の中身を説明すること
- 抽象的なコピーだけでページを成立させること
- 同じ説明を全ページで繰り返すこと
- 見出し構造が曖昧なこと
- 1ページ内に論点を詰め込みすぎること
- 一覧と詳細の役割が混ざること

---

## GEOを意識した運用方針

### 更新時の確認事項

| 更新対象 | 確認ポイント |
|:---|:---|
| 全ページ共通 | そのページは何のページか / 何が分かるか / 誰向けか が明確か |
| 実績追加時 | 案件概要・課題・対応範囲・実施内容・成果を省略していないか |
| サービス更新時 | 対象顧客・課題・提供内容・対応範囲が曖昧になっていないか |
| 料金更新時 | レンジ変更だけでなく、説明文との整合が取れているか |
| お問い合わせ更新時 | 入力しやすさ・相談可能範囲・送信後の流れが保たれているか |

## GEO改善の観点

後続運用で確認すべき観点：

- 生成AIでブランド名やサービス名を質問した際に、何が抽出されるか
- ページ冒頭だけで内容理解できるか
- ページごとに要約可能な固有情報があるか
- 案件ページが案件ごとに差別化されているか
- 価格ページが価格だけでなく考え方も伝えているか
- 問い合わせページが相談可能範囲を明示しているか
