# SocialBoost 公式Webサイト 構造化データ設計書

---

## 文書目的

本書は、SocialBoost 公式Webサイトにおいて、検索エンジンおよび生成AIがページ内容、事業情報、サービス情報、実績情報、FAQ情報、パンくず情報を正しく理解しやすくするための構造化データ方針を定義することを目的とする。
本書は、SEO設計書およびGEO設計書を補完し、後続の実装設計、テスト設計、運用設計の基準文書とする。

---

| 項目 | 内容 |
|:---|:---|
| 名称 | SocialBoost 公式Webサイト |
| プロジェクトフォルダ | `socialboost` |

## 前提

本フェーズでは実装を行わない。
本書は構造化データの設計文書であり、JSON-LD実装、head設定、metadata設定、検証ツール実行、公開作業、デプロイは行わない。
ユーザー本人が明示的に実装許可を出すまで、実装フェーズへ進まない。

---

## 構造化データ設計の基本思想

SocialBoost のサイトは、単なる制作会社サイトとしてではなく、Web制作、Webシステム開発、アプリ開発、AI導入支援、改善運用支援まで対応できる事業成長パートナーとして理解される必要がある。
構造化データは見た目の装飾ではなく、事業情報、ページ役割、ページ間関係、サービス内容、FAQ、パンくずを明確化するための基盤として扱う。

---

## 構造化データ設計の目的

1. 検索エンジンにページ種別を正しく伝えること
2. ブランド情報とサービス情報を誤解なく伝えること
3. サービス詳細ページを独立したサービス説明ページとして理解させること
4. FAQ的情報やパンくずを整理して回遊性を補強すること
5. GEOの観点からも、何のページかを機械的に読み取りやすくすること
6. 将来的な拡張ページにも一貫したルールを適用できるようにすること

---

## 基本方針

1. JSON-LD を基本採用方針とする
2. ページ種別ごとに必要なスキーマを整理する
3. 実際のページ内容と一致しない情報は記述しない
4. 存在しないページ、存在しない機能、誇張した事業範囲は記述しない
5. 一覧ページと詳細ページで役割の違う構造化データを使い分ける
6. 同一ページに複数スキーマを入れる場合は、役割と整合性を崩さない
7. 将来的なFAQページ、資料ページ、記事ページ追加に備えた拡張性を持つ

---

## 構造化データの優先対象ページ

| 優先度 | 対象 |
|:---|:---|
| 最優先 | トップページ、サービス詳細ページ、料金ページ、実績詳細ページ、お問い合わせページ |
| 優先 | サービス一覧ページ、実績一覧ページ、選ばれる理由ページ、プライバシーポリシーページ、将来のFAQページ |
| 後続拡張 | 資料一覧/詳細ページ、記事一覧/詳細ページ、カテゴリページ、法務ページ群 |

## ページ種別ごとの基本スキーマ方針

| ページ | 主候補スキーマ |
|:---|:---|
| トップページ | Organization / WebSite / BreadcrumbList / WebPage |
| サービス一覧 | WebPage / BreadcrumbList / ItemList |
| サービス詳細 | Service / WebPage / BreadcrumbList |
| 料金ページ | WebPage / BreadcrumbList / FAQPage（部分適用余地） |
| 実績一覧 | WebPage / BreadcrumbList / ItemList |
| 実績詳細 | WebPage / BreadcrumbList / CreativeWork（慎重に検討） |
| 選ばれる理由 | WebPage / BreadcrumbList / FAQPage（部分適用余地） |
| お問い合わせ | ContactPage / WebPage / BreadcrumbList |
| プライバシーポリシー | WebPage / BreadcrumbList |

---

## トップページの構造化データ設計

### 採用候補スキーマ

Organization / WebSite / WebPage / BreadcrumbList

### Organization で表現すべき情報

- 名称、ブランド名、ロゴ、URL
- 事業概要
- 問い合わせ先の基本情報
- 対応領域の概要

### 記載方針

- 法人格を誤認させる表現は避ける
- 現時点の事業実態に合わせる
- 個人事業主段階であることと矛盾しない範囲で整理する
- 所在地や連絡先の扱いは実際に公開する情報範囲に合わせる

### WebSite で表現すべき情報

- サイト名、URL、ブランド全体の説明
- 検索機能を将来持つ場合の拡張余地

---

## サービス一覧ページの構造化データ設計

### 採用候補スキーマ

WebPage / BreadcrumbList / ItemList

### ItemList の対象

サービス一覧カード群。各項目にサービス名、詳細URL、簡潔な説明を持たせる。

### 方針

- 一覧ページでは一覧の役割に徹し、各サービスの詳細説明は詳細ページへ委譲する
- 各サービス項目が詳細ページと紐づくことを明確にする

---

## サービス詳細ページの構造化データ設計

### 採用候補スキーマ

Service / WebPage / BreadcrumbList

### Service で表現すべき情報

- サービス名、説明、提供者
- 対応範囲の要約
- 対象ユーザーの要約
- 関連URL
- 必要に応じたサービスカテゴリ

### 記載方針

- ページ本文と一致する内容のみ記載する
- 実際に存在しない実績数や成果を記載しない
- 価格情報を持たせる場合は実態と整合する場合に限定し、初期段階では慎重に扱う

### BreadcrumbList

トップ → サービス一覧 → 対象サービス詳細

---

## 料金ページの構造化データ設計

### 採用候補スキーマ

WebPage / BreadcrumbList / FAQPage（部分利用余地）

### 方針

- 初期段階では、料金ページを無理に Product や Offer として表現しない
- 価格が参考レンジであり一律商品ではないため、WebPage として扱う
- FAQ的な不安解消が十分に整理される場合は FAQPage の適用余地を持つ

---

## 実績一覧ページの構造化データ設計

### 採用候補スキーマ

WebPage / BreadcrumbList / ItemList

### ItemList の対象

実績カード一覧。各項目に案件名、詳細URL、カテゴリ、簡潔な説明、必要に応じて画像を持たせる。

### 方針

- 一覧ページでは一覧の構造を伝えることを優先する
- 案件ごとの本文情報は詳細ページで扱う

---

## 実績詳細ページの構造化データ設計

### 採用候補スキーマ

WebPage / BreadcrumbList / CreativeWork（慎重に検討）

### 基本方針

- 初期段階では案件ページを無理に Article や Product として扱わない
- CreativeWork 系の適用可否は案件本文の実態に応じて慎重に検討する
- まずは WebPage として、案件概要、課題、対応範囲、実施内容、成果がある構造として整理する

### BreadcrumbList

トップ → 実績一覧 → 対象案件詳細

### 画像とPDFに関する方針

- 画像やPDFはページ本文の補助であり、初期段階ではそれ自体を独立スキーマ主体にしない
- 必要に応じて image 情報を補助的に持たせる余地を残す

---

## 選ばれる理由ページの構造化データ設計

### 採用候補スキーマ

WebPage / BreadcrumbList / FAQPage（部分適用余地）

### 方針

- 比較検討に関する要点整理ページとして扱う
- 強みや進め方がFAQ形式に整理される場合、FAQPage の適用を検討する
- 質問と回答の構造が明確であることが適用条件

---

## お問い合わせページの構造化データ設計

### 採用候補スキーマ

ContactPage / WebPage / BreadcrumbList

### ContactPage で表現すべき情報

- ページ名、問い合わせ目的
- 相談可能範囲の要約
- 連絡方法の基本情報
- 必要に応じた組織情報との接続

### 方針

- フォーム実態と一致すること
- 誤解を招く連絡手段を記載しないこと

---

## プライバシーポリシーページの構造化データ設計

### 採用候補スキーマ

WebPage / BreadcrumbList

特別なスキーマを無理に付与しない。ページとしての役割を明確にする。

---

## FAQ系構造化データの設計方針

### FAQPage を検討する対象

- 料金ページの不安解消セクション
- 選ばれる理由ページのFAQ的要素
- お問い合わせページの相談可否や送信前の不安解消
- 将来のFAQ専用ページ

### 適用条件

- 質問と回答の形で明確に分かれていること
- 本文内にも同様の情報が確認できること
- ページ内容と整合していること
- 検索エンジンガイドラインに反しないこと

### FAQ候補例

- 要件が固まっていなくても相談できますか
- 小規模案件でも対応できますか
- 料金はどのように決まりますか
- 制作だけでなくシステムやアプリも相談できますか
- 公開後の改善も依頼できますか

### 適用方針

- FAQを増やしすぎない
- 1ページ内で役割が曖昧になるほど詰め込まない
- ユーザーにとって意味のある質問と回答に限定する

---

## BreadcrumbList 設計方針

### 基本階層

| ページ | 階層 |
|:---|:---|
| サービス詳細 | トップ → サービス → 各サービス |
| 実績詳細 | トップ → 実績 → 各実績 |
| 資料詳細（将来） | トップ → 資料 → 各資料 |
| 記事詳細（将来） | トップ → ブログ → 各記事 |

### 方針

- 実際のサイト構造と一致させる
- 存在しない親ページを含めない
- パンくず表示UIと整合することが望ましい

---

## Organization 設計方針

### 記載対象候補

名称 / URL / ロゴ / 説明 / 問い合わせ先 / SNSや関連リンク / 所在地（公開する場合に限る）

### 注意点

- 法人格を誤認させる情報を載せない
- 存在しない拠点や電話窓口を記載しない
- 実際の公開方針と一致させる

---

## Service 設計方針

### 記載対象候補

サービス名 / 説明 / 提供者 / 対象顧客の要約 / 対応範囲の要約 / 関連URL / 必要に応じた areaServed や serviceType

### 注意点

- 価格の扱いは慎重にする
- 曖昧な定量表現を避ける
- 本文に存在しないサービス範囲を追加しない

---

## ItemList 設計方針

### 対象ページ

サービス一覧 / 実績一覧 / 将来の資料一覧 / 将来のブログ一覧

### 方針

- 一覧ページでは ItemList の有効性が高い
- 項目数が多くなる場合の扱いは後続技術設計で制御する前提

---

## 構造化データと本文の整合ルール

- 本文にない内容を構造化データへ追加しない
- 実績数、対応範囲、料金、連絡方法などは実態と一致させる
- ページ名、見出し、description、構造化データの意味を揃える
- FAQ構造は本文内にも質問と回答が存在することを前提とする

---

## 構造化データの実装単位方針

- ページごとに必要な最小構成から始める
- 一度に全スキーマを詰め込まない
- まずはトップページ、サービス詳細、実績詳細、お問い合わせを優先する
- 一覧ページやFAQ適用は後続段階で拡張してよい

---

## JSON-LD 運用方針

JSON-LD を第一候補とする理由：  
ページ構造と分離しやすい / 保守性が高い / ページ単位で制御しやすい / 後続の技術設計と相性がよい

---

## 運用更新方針

| 更新頻度 | 対象 |
|:---|:---|
| 高頻度 | サービス詳細の説明、実績詳細の案件本文、FAQ項目、問い合わせの相談範囲、価格説明文 |
| 中頻度 | Organization情報、一覧ItemList、BreadcrumbList |
| 低頻度 | サイト全体方針、スキーマ採用方針、ルール定義 |

### 更新ルール

- 本文更新時は構造化データとの整合を確認する
- 実績追加時は一覧、詳細、パンくず、説明要約の整合を確認する
- 使っていないスキーマを形だけ残さない

---

## 検証前提

後続実装時に確認すべき検証観点：

- ページ種別ごとに適切なスキーマが適用されているか
- ページ本文と矛盾していないか
- FAQPage の適用対象が適切か
- BreadcrumbList が実際の階層と一致しているか
- Service の説明がサービスページ本文と整合しているか
- Organization 情報が実際の公開情報と一致しているか

---

## 構造化データ設計で避けること

- Product や Offer を無理に多用すること
- 法人格を誤認させる Organization 情報
- 存在しない価格の断定記載
- 本文にない FAQ の追加
- 画像やPDFだけに依存する説明
- 一覧ページと詳細ページの役割混同
- 検索結果表示を狙っただけの過剰スキーマ追加

---

## 将来拡張方針

将来的に追加検討可能なスキーマ：  
FAQPage / Article / CollectionPage / CreativeWork / Document  
ただし、その時点のページ実態とガイドライン適合を確認してから適用する。
