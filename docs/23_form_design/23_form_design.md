# SocialBoost 公式Webサイト フォーム設計書

---

## 文書目的

本書は、SocialBoost 公式Webサイトにおけるお問い合わせフォームの設計方針、入力項目、入力体験、エラーハンドリング、送信フロー、セキュリティ配慮、ユーザー心理への配慮を定義することを目的とする。
本書は、コピーライティング設計書、内部リンク設計書、SEO設計書、GEO設計書、UX設計書を補完し、後続の実装設計、テスト設計、運用設計の基準文書とする。

## 前提

本フェーズでは実装を行わない。
本書はフォームの設計文書であり、実際のフォーム実装、API実装、メール送信処理、データ保存処理、デプロイは行わない。
ユーザー本人が明示的に実装許可を出すまで、実装フェーズへ進まない。

---

## フォーム設計の基本思想

SocialBoost のお問い合わせフォームは「営業フォーム」ではなく**「相談フォーム」**として設計する。
ユーザーが問い合わせる時点では要件が整理されていないケースが多いため、入力の負担を減らし、安心感を与え、途中離脱を防ぐことを優先する。

---

## フォーム設計の目的

1. 相談の心理的ハードルを下げる
2. 必要な情報だけを収集する
3. 入力ストレスを減らす
4. 問い合わせ送信率を高める
5. スパムや不正送信を防ぐ
6. 将来的なCRMや案件管理に接続できる設計にする

---

## フォームの役割

- ユーザーの相談内容を受け取る
- 問い合わせ内容を整理する
- 折り返し連絡のための情報を取得する
- 案件の概要を把握する

---

## フォーム配置ページ

入力フォーム自体は**お問い合わせページに集約**する。
※以下の箇所からは、お問い合わせページへのCTA導線のみを配置する。
- トップページ下部CTA
- サービス詳細ページCTA
- 実績詳細ページCTA
- 料金ページCTA

---

## フォーム基本構造

お問い合わせフォームは以下の要素で構成する。

1. 問い合わせカテゴリ
2. 会社名または屋号（任意）
3. 担当者名（必須）
4. メールアドレス（必須）
5. 電話番号（任意）
6. 相談内容（必須）
7. プライバシーポリシー同意（必須）

---

## 各入力項目の設計

### 1. 問い合わせカテゴリ

| 項目 | 内容 |
|:---|:---|
| 役割 | 問い合わせ内容の分類、初期対応の優先順位整理 |
| 選択肢候補 | Web制作 / LP制作 / Webシステム開発 / アプリ開発 / AI導入 / 料金 / その他 |
| 選択方式 | セレクトボックス または ラジオボタン |

### 2. 会社名または屋号

| 項目 | 内容 |
|:---|:---|
| 目的 | 法人問い合わせの識別、個人問い合わせにも対応するため |
| 必須/任意 | **任意** |
| 入力方式 | テキスト入力 |
| 理由 | 個人事業主や個人からの相談の可能性を考慮するため |

### 3. 担当者名

| 項目 | 内容 |
|:---|:---|
| 目的 | 連絡先の識別 |
| 必須/任意 | **必須** |
| 入力方式 | テキスト入力 |
| 入力補助例 | 山田 太郎 |

### 4. メールアドレス

| 項目 | 内容 |
|:---|:---|
| 目的 | 返信先 |
| 必須/任意 | **必須** |
| 入力方式 | メール入力 |
| 入力チェック | メール形式チェック |
| 入力補助例 | 折り返しご連絡するメールアドレスをご入力ください。 |

### 5. 電話番号

| 項目 | 内容 |
|:---|:---|
| 目的 | 緊急連絡や詳細確認 |
| 必須/任意 | **任意** |
| 入力方式 | 電話番号入力 |
| 理由 | 電話連絡を望まないユーザーの心理的ハードルを下げるため |

### 6. 相談内容

| 項目 | 内容 |
|:---|:---|
| 目的 | 案件概要の把握 |
| 必須/任意 | **必須** |
| 入力方式 | テキストエリア |
| 入力補助例 | 相談したい内容や検討していることを、分かる範囲でご記入ください。要件が整理されていない状態でも問題ありません。 |

### 7. プライバシーポリシー同意

| 項目 | 内容 |
|:---|:---|
| 役割 | 法務対応 |
| 必須/任意 | **必須** |
| 形式 | チェックボックス |
| 文言 | プライバシーポリシーに同意する |
| リンク先 | プライバシーポリシーページへのリンク |

## 入力UX設計

フォームは以下のUX原則に従う。

1. 入力項目は最小限にする
2. 必須項目を明確にする
3. エラーは入力後すぐに表示する（リアルタイムバリデーション）
4. モバイルでも入力しやすいUIにする
5. プレースホルダや補助テキストで入力例を提示する
6. フォーム周りの長文説明を避ける
7. CTAボタンは視覚的に目立たせる

---

## エラー表示設計

エラーはユーザーを責めない優しい表現とする。

| エラー内容 | 推奨表示例 |
|:---|:---|
| メール形式不正 | メールアドレスの形式をご確認ください |
| 必須項目未入力（内容） | 相談内容を入力してください |
| 必須項目未入力（名前） | 担当者名を入力してください |

**禁止表現（強い否定表現）：**
- 入力が間違っています
- 正しい値を入力してください

---

## 送信ボタン設計

| 項目 | 内容 |
|:---|:---|
| 推奨文言 | **相談内容を送信する** |
| 理由 | 営業感を弱め、相談フォームという位置づけに合わせるため |
| その他の候補 | お問い合わせを送信する / 相談してみる |

---

## 送信後フロー

送信完了後、以下のメッセージを表示する構成（画面遷移または同画面内での状態変化）とする。

**送信完了メッセージ設計：**
> お問い合わせを受け付けました。
> 内容を確認のうえ、折り返しご連絡いたします。
> 必要に応じて、進め方や予算感の整理も含めてご案内します。

---

## スパム対策設計

スパム・不正送信を防ぐため、以下の対策を検討する。
※具体的な実装方式は技術設計書にて定義する。

1. reCAPTCHA
2. Honeypotフィールド
3. 送信回数制限
4. IP制限
5. メールドメインチェック

---

## セキュリティ配慮

以下のセキュリティ対策を講じる。
※具体的な実装方式は技術設計書にて確定する。

- 入力データの適切なサニタイズ
- SQLインジェクション対策
- XSS（クロスサイトスクリプティング）対策
- CSRF（クロスサイトリクエストフォージェリ）対策
- 送信ログの安全な管理（必要最小限の期間のみ保持等）

---

## 問い合わせ対応フロー

本フォームを経由した業務フローは以下の通りとする。（詳細は運用設計書と連携）

1. 問い合わせ送信
2. 通知メール送信（管理者宛、控えとしてユーザー宛）
3. 内容確認
4. 返信・ヒアリング
5. 案件化 または 相談終了

## SEOおよびGEO観点の設計

### GEO観点
AIがページ内容を理解しやすくするため、問い合わせページでは以下を明確に構造化された文章で提示する。
- 相談できる内容
- 入力項目
- 返信方法
- 対応範囲

### SEO観点
お問い合わせページはコンバージョンを目的とするページであり、過剰なSEOキーワード挿入は行わない。ただし、ページの meta description 等で「何が相談できるか」の説明は明確にする。

---

## 運用ルール

- 問い合わせ内容は定期的に分析する
- フォームの入力途中離脱率を確認する
- 必要に応じて入力項目（カテゴリの増減など）を調整する
- 送信される問い合わせの質（スパム、的外れな相談がないか等）を確認し、改善に役立てる
