# SocialBoost 公式Webサイト セキュリティ設計書

---

## 文書目的

本書は SocialBoost 公式Webサイトのセキュリティ設計方針を定義する文書である。
Webサイトにおける情報漏洩、改ざん、不正アクセス、スパム、攻撃リスクを最小化することを目的とする。

本書は以下の設計書と連携する。
- 要件定義書
- 基本設計書
- 詳細設計書
- フォーム設計書
- 技術設計書
- 運用設計書
- テスト設計書

本書では「設計レベル」のセキュリティ方針を定義する。
実際の実装はユーザーの許可が出るまで行わない。

## セキュリティ設計の基本方針

1. **最小権限の原則**
   必要最小限のアクセス権のみを許可する。
2. **多層防御**
   単一対策ではなく複数のセキュリティレイヤーで防御する。
3. **入力値を信頼しない**
   すべてのユーザー入力は不正データの可能性がある前提で処理する。
4. **公開情報と内部情報の分離**
   内部情報は公開環境に置かない。
5. **自動化された攻撃を想定**
   Bot、スパム、スクレイピング対策を前提とする。

## Webサイト公開レイヤーのセキュリティ

### HTTPSの強制
すべての通信はHTTPSを使用する。
HTTPアクセスがあった場合は自動的にHTTPSへリダイレクトする。

### セキュアヘッダー
以下のセキュリティヘッダーを設定する。
- `Content-Security-Policy`（XSS攻撃軽減、外部スクリプトの制御、インラインスクリプト制御、外部リソースのホワイトリスト化）
- `X-Frame-Options`（クリックジャッキング防止）
- `X-Content-Type-Options`（MIMEタイプ攻撃防止）
- `Referrer-Policy`（安全なリファラー制御）
- `Strict-Transport-Security`

---

## フォームセキュリティ

フォームは最も攻撃対象になりやすいため、特に対策を行う。

### 入力値検証
すべての入力項目は以下を行う。
- 文字数制限
- 文字種制限
- 不正コード検出
- サニタイズ処理

### XSS対策
- HTMLタグの除去
- スクリプトタグの排除
- HTMLエスケープ処理

### CSRF対策
- CSRFトークンを使用する。
- フォーム送信時にトークン検証を行う。

### SQLインジェクション対策
- 直接SQLを組み立てない。
- ORMまたは安全なクエリを使用する。

### スパム対策
以下の仕組みを導入する。
- Honeypotフィールド
- reCAPTCHA
- 送信回数制限
- IPレート制限
