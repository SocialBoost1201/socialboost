## リリース設計の基本方針

1. **手動リリース禁止**
   直接本番サーバーを変更しない。
2. **Gitベース管理**
   すべての変更はGit管理とする。
3. **自動デプロイ**
   CI/CDで自動反映する。
4. **Preview検証**
   本番反映前に必ずPreview環境で確認する。
5. **段階公開**
   テスト完了後にのみ公開する。

## デプロイ環境構成

`GitHub → Vercel → Production`

### ブランチ運用

| ブランチ名 | 用途 |
|:---|:---|
| `main` | 本番ブランチ（Production環境） |
| `feature` | 開発ブランチ（Preview環境生成用） |

## デプロイ・リリースフロー

### デプロイフロー

```text
開発
  ↓
Git commit
  ↓
GitHub push
  ↓
Vercel build
  ↓
Preview環境
  ↓
確認
  ↓
Production反映
```

### リリースフロー

```text
開発完了
  ↓
Pull Request
  ↓
Preview生成
  ↓
確認
  ↓
mainマージ
  ↓
本番デプロイ
```

---

## Preview環境運用

Vercel Previewを使用する。

**用途：**
- デザイン確認
- UI確認
- リンク確認
- SEO確認

---

## 公開前チェック

リリース（`main`マージ）前に以下の項目をPreview環境にて確認する。

- ページ表示
- リンク切れ
- フォーム送信
- SEO設定
- OGP表示
- モバイル表示
- パフォーマンス

---

## 公開手順

1. `main`ブランチ更新（マージ）
2. Vercel build開始
3. ビルド成功確認
4. 本番URL反映

---

## デプロイ確認（本番確認）

本番公開後に以下の動作を必ず確認する。

- トップページ表示
- 各ページ表示
- リンク動作
- フォーム送信
- 画像表示

---

## ロールバック設計

問題発生時は直前バージョンに戻す。

**方法：**
```text
GitHub revert
  ↓
再デプロイ
```

---

## 障害・緊急対応

重大障害発生時（サイト停止等）のフロー：

```text
サイト停止
  ↓
原因調査
  ↓
修正
  ↓
再デプロイ
```
