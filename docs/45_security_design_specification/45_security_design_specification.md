# SocialBoost 公式Webサイト セキュリティ設計書

---

## 文書目的

本書は SocialBoost 公式Webサイトにおけるセキュリティ方針および具体的対策を定義する。

本サイトは以下の情報を扱う。
- 問い合わせ情報
- 企業情報
- 制作実績データ
- PDF資料

そのため以下のセキュリティ対策を実装前提とする。
- 通信の安全性
- データ保護
- 不正アクセス防止
- スパム対策
- アプリケーションセキュリティ

本書ではシステム全体のセキュリティ設計を定義する。

## セキュリティ設計方針

1. **最小権限原則**: システムおよびユーザーに必要最低限の権限のみ付与する。
2. **入力データの検証**: ユーザーからのすべての入力データを厳密に検証（サニタイズ・バリデーション）する。
3. **HTTPS通信**: 通信経路はすべて暗号化し、盗聴等を防ぐ。
4. **機密データの保護**: データベース等における情報の安全管理を徹底する。

---

## 通信セキュリティ
- **HTTPS必須**: すべての通信（フロントエンド、APIアクセス等）はHTTPSを使用する。
- **SSL証明書**: Vercelの自動SSL適用機能を利用して暗号化を担保する。

---

## アプリケーションセキュリティ

### 入力バリデーション
- **対象**: 問い合わせフォーム等すべてのユーザー入力
- **チェック項目**: 必須入力確認、文字数制限（最大/最小）、メール形式確認

### XSS（クロスサイトスクリプティング）対策
- **対策**: HTMLエスケープ処理の徹底
- React (Next.js) の基本機能に加えて、任意のユーザー入力文字列を出力する際のサニタイジングを行う。

### CSRF（クロスサイトリクエストフォージェリ）対策
- Next.js Server Functions の標準機能を活用して不正なリクエストを防ぐ。

### SQLインジェクション対策
- **対策**: ORMやクエリビルダーの使用
- リレーショナルDBの直接操作は避け、**Supabaseクエリ** 等の安全にパラメータ化された機能を使用する。

---

## スパム・レート制限対策

### スパム対策
- `Google reCAPTCHA`（v3等の不可視型を推奨）を導入し、ボットによる自動送信を防ぐ。

### レート制限（Rate Limiting）
- **対象**: 問い合わせAPI送信
- **制限内容**: 同一IP等につき `1分間に5回まで` とする。

---

## アクセス制御・認証

- **管理機能（将来構想含む）**: 保護されたエンドポイントやCMS機能へのアクセスには認証を必須とする。
- **採用技術**: `Supabase Auth` を使用しセキュアな認証状態管理を行う。

---

## データ保護保護とストレージ

### 情報保護とログ
- **問い合わせ情報**: 安全にSupabase DBへ保存する。
- **ログ対象**: APIアクセス、エラー、問い合わせ送信など追跡可能な情報
- **保存管理**: Supabase等に適切に記録し、管理機能以外からのアクセスを遮断する。

### ストレージセキュリティ
- **画像・PDF保存**: `Supabase Storage` に保存し、必要なアセットは公開URL（Public URL）を通じて配信し、機密ファイルは認証付きURLとして管理する。

---

## インフラ・運用セキュリティ

### 環境変数設定（機密情報管理）
- **対象**: 各種APIキー、DB接続URLや認証キー等
- **保存・管理**: コード内へのハードコードは厳禁とし、**Vercel Environment Variables**（サーバー側の環境変数）として管理する。

### エラー情報の隠蔽化
- 本番環境でのアプリケーションエラー・詳細情報はユーザーインターフェースに表示させない。フロントエンドには一般的なエラーメッセージを返し、詳細ログはサーバー側（ログ機構）のみに保存する。

### バックアップと脆弱性対策
- **バックアップ**: Supabaseの自動バックアップ機能（PITR等）を活用する。
- **脆弱性対策（日々の保守）**: 依存パッケージやフレームワークの定期的なセキュリティアップデートを行う。
- **監視**: エラーログおよびアクセスログの継続的な監視体制（Vercel Analytics等との連携）を設ける。

## 本書で確定する事項

| No. | 決定事項 |
|:---|:---|
| 1 | すべての通信でHTTPSを必須とする（Vercel SSL） |
| 2 | 厳格な入力バリデーション、XSS・CSRF・SQLインジェクション対策を実装する |
| 3 | スパム対策として `reCAPTCHA` を導入し、APIにはレート制限を課す |
| 4 | 管理機能におけるアクセス制御は `Supabase Auth` によって保護する |
| 5 | 機密情報（APIキー等）は環境変数（Vercel）で管理・隠蔽する |
| 6 | ユーザーにシステムエラーの内部詳細を露出させない設計を徹底する |
| 7 | 現時点では実装（コード記述、設定追加等）は行わない |
