# SocialBoost 公式Webサイト UX設計書

---

## 文書目的

本書は SocialBoost 公式Webサイトにおけるユーザー体験設計の方針、導線設計、意思決定支援設計、離脱防止設計、コンバージョン最適化方針を定義する文書である。

本サイトは単なる情報掲載サイトではなく、問い合わせ獲得と信頼形成を目的とする **営業導線型サイト** であるため、見た目の美しさだけではなく、理解しやすさ、比較しやすさ、相談しやすさを一貫して設計する必要がある。

本書は以下を目的とする。
- 何をしている事業かを短時間で理解させる
- ユーザーが自分に必要な情報へ迷わず辿り着けるようにする
- 問い合わせ前の不安を減らす
- 比較検討時に信頼材料を整理して提示する
- 実績、料金、サービス、問い合わせを自然につなぐ

本書は以下の設計書と連携する。
- 要件定義書 / 基本設計書 / 詳細設計書
- 情報設計書 / UI設計書
- トップページ設計書 / サービスページ設計書 / 料金ページ設計書 / 実績ページ設計書 / お問い合わせページ設計書
- SEO設計書 / GEO設計書 / 分析・データ設計書

## UX設計の基本思想

ユーザーの前提認識：
- 最初から要件が固まっているとは限らない
- 制作物そのものより、自分の課題が解決できるかを見ている
- 価格だけでなく、信頼、実績、進めやすさを見ている
- 営業されたいのではなく、相談できる相手を探している
- UXは派手な動きではなく、迷わない体験によって成立する

### UX基本原則
1. **理解しやすい**
2. **迷わない**
3. **比較しやすい**
4. **相談しやすい**
5. **安っぽく見えない**
6. **スマホでもストレスがない**

---

## UX設計の目的

1. 初回訪問時の理解負荷を下げる
2. サービスの全体像と個別内容を整理して伝える
3. 制作だけでなく、システム、アプリ、AIまで対応できることを自然に理解させる
4. 実績を通じて信頼を形成する
5. 料金への不安を適切に下げる
6. 問い合わせの心理的ハードルを下げる
7. 離脱を減らし、必要な下層ページへ送る
8. 問い合わせまでの最短導線と、比較検討のための回遊導線を両立する

---

## 対象ユーザー整理

### 主要ユーザー
- 中小企業経営者
- 個人事業主
- 店舗事業者
- 新規事業立ち上げ担当者
- 既存サイトや導線改善を検討している担当者
- システム化、アプリ化、AI活用を検討している担当者

### 想定される状態
- 何を頼めばいいか分からない
- 依頼内容はある程度あるが整理できていない
- 制作会社との違いが分からない
- 実績や価格感を見てから相談したい
- 小規模でも相談してよいか不安
- 相談後の流れが見えず不安

この前提から、本サイトのUXは「明確な提案型」ではなく **「整理支援型」** の体験を優先する。

---

## UX上の主要課題

1. トップページで何をしている事業かが抽象的だと離脱しやすい
2. サービスが多いと何を見ればいいか分からなくなる
3. 価格だけ見るユーザーは高いか安いかだけで判断しやすい
4. 実績一覧だけでは中身が伝わらず、信頼形成が弱くなる
5. 問い合わせページが入力負荷過多だと送信率が落ちる
6. スマホでの読みづらさ、押しづらさがCVRを下げる
7. 強い営業感は個人事業主サイトに対する警戒感を高める

---

## UX設計の成功条件

- 何の事業かがすぐ分かる
- 自分に関係あるサービスが見つかる
- 実績を見て相談イメージが湧く
- 価格の考え方に納得できる
- 問い合わせしても大丈夫そうだと感じる
- スマホでストレスがない
- サイト全体が整理されていて信頼できる

## ユーザージャーニー設計

### 初回流入フェーズ
- **流入元:** 自然検索、紹介、SNS、名刺や営業資料、ポートフォリオ共有
- **ユーザー心理:** どんな会社か分からない / 頼める範囲が広いのか知りたい / デザインレベルと信頼感を見たい
- **必要UX:** トップで事業内容が分かる / 主要サービスが把握できる / 実績導線が明確 / 問い合わせが押し売りに見えない

### 比較検討フェーズ
- **ユーザー心理:** 他社と何が違うのか知りたい / 制作だけか、それ以上できるのか知りたい / 価格感と実績を見たい
- **必要UX:** 選ばれる理由セクション / サービス詳細ページ / 料金ページ / 実績詳細ページ / 問い合わせへの自然な橋渡し

### 相談判断フェーズ
- **ユーザー心理:** 小規模でも相談してよいか / まだ整理できていなくてもよいか / 価格の相談もできるか
- **必要UX:** 不安解消文言 / 相談歓迎の文脈 / 入力項目の少ないフォーム / 送信後の流れが見えること

## 主要ユーザーフロー

| フロー | 導線 |
|:---|:---|
| **A** | トップページ → サービス一覧 → サービス詳細 → 関連実績 → お問い合わせ |
| **B** | トップページ → 実績一覧 → 実績詳細 → 料金ページ → お問い合わせ |
| **C** | 自然検索 → サービス詳細 → 料金ページ → お問い合わせ |
| **D** | 自然検索 → 実績詳細 → 関連実績 → お問い合わせ |
| **E** | トップページ → 選ばれる理由 → 実績一覧 → お問い合わせ |

UX設計上、この複数導線を同時に成立させる必要がある。

## ページ別UX役割

### トップページ
- **役割:** 全体理解の入口、第一印象形成、主要導線への分岐点
- **UX要件:** 何をしているかを一瞬で伝える / 主要サービスが理解できる / 実績を見たくなる / 問い合わせが押し売りに見えない / 情報を詰め込みすぎない

### サービス一覧ページ
- **役割:** サービス比較、課題からのサービス特定
- **UX要件:** 一覧で比較しやすい / 説明しすぎない / 詳細ページへ自然に進める / 依頼内容が曖昧でも選びやすい

### サービス詳細ページ
- **役割:** 個別理解、相談判断
- **UX要件:** 誰向けかが分かる / 課題との接続が明確 / 提供内容と対応範囲が分かる / 関連実績と料金導線がある / 問い合わせへの心理的接続が自然

### 料金ページ
- **役割:** 価格不安解消、誤解防止、相談導線強化
- **UX要件:** 数字だけでなく考え方が分かる / 高圧的でない / 安売りに見えない / 小規模相談も可能だと分かる / サービス理解へ戻れる

### 実績一覧ページ
- **役割:** 品質感の提示、比較閲覧、詳細ページへの入口
- **UX要件:** サムネイル主役 / 一覧性が高い / カテゴリや業種の違いが分かる / 詳細を見たくなる / スマホでも見やすい

### 実績詳細ページ
- **役割:** 案件理解、信頼形成、営業資料代替
- **UX要件:** 案件概要がすぐ分かる / 課題・対応範囲・実施内容が整理されている / 画像とPDFが混在しない / 問い合わせのきっかけになる / 関連実績に回遊できる

### お問い合わせページ
- **役割:** 最終行動ページ、相談受付
- **UX要件:** 入力負荷が低い / 要件未整理でも相談可能と分かる / 送信後の流れが見える / 法務導線が自然 / 補助リンクはあるが主役はフォーム

## 理解負荷低減設計

### 基本方針
- トップでは要約だけを見せる
- 一覧では比較に必要な情報だけを見せる
- 詳細では固有情報を整理して見せる
- ページの役割を混ぜない

### 理解負荷を下げる手段
- 見出しで意味を即時伝える
- 長文を分割する
- カード化で一覧性を高める
- 課題ベースで説明する
- 似た説明の重複を減らす
- 画像と本文を役割分担する

### 避けるべきUX
- トップページが長すぎて、読了前に何の会社か分からない
- サービス一覧に長文が並び比較できない
- 実績一覧で情報が多すぎて散らかって見える
- 料金ページが数字だけで冷たい印象になる
- 問い合わせフォームが重くて離脱する

## 意思決定支援UX設計

ユーザーは「見た目がよい」だけでは動かないため、意思決定に必要な材料を順番に与える。

| 必要材料 | 配置先 |
|:---|:---|
| **何が頼めるか** | トップページ / サービス一覧 / サービス詳細 |
| **自分に合うか** | サービス詳細の対象顧客 / よくある課題 |
| **どんな実績があるか** | 実績一覧 / 実績詳細 |
| **どのくらいの価格感か** | 料金ページ / トップページの料金案内 |
| **どう進むか** | トップページの進行フロー / サービス詳細ページ |
| **相談しても大丈夫か** | 不安解消セクション / お問い合わせページ |

## コンバージョンUX設計

### コンバージョンの定義
本サイトにおける主要CVは **お問い合わせ送信** とする。

### 中間CV候補
- サービス詳細閲覧
- 料金ページ閲覧
- 実績詳細閲覧
- お問い合わせページ到達

### CVRを高めるUX原則
- 問い合わせ前に不安を解消する
- 価格の誤解を防ぐ
- 実績による信頼形成を挟む
- CTAをページ文脈に合わせる
- 押し売り感を避ける

### ページ別主要CTA

| ページ | CTA文言例 |
|:---|:---|
| **トップページ** | サービスを見る / 実績を見る / お問い合わせする |
| **サービス詳細** | この内容で相談する / 料金を見る |
| **料金ページ** | 予算感を踏まえて相談する / サービス内容を見る |
| **実績詳細** | このような案件を相談する / 関連実績を見る |
| **問い合わせページ** | 相談内容を送信する |

### CTAのUX要件
- 文言だけで遷移先が分かる
- ページ文脈と合う
- 同一画面で優先順位が曖昧にならない
- スマホで押しやすい

## 離脱防止UX設計

### 主な離脱要因と対策

| 離脱要因 | 対策 |
|:---|:---|
| 何の事業か分からない | トップで一文定義を置く |
| 情報が多すぎる | 一覧ページは短文化する |
| 価格だけ見て判断できない | 価格ページで考え方を説明する |
| 信頼材料が足りない | 実績詳細で固有情報を出す |
| 問い合わせが重い | 問い合わせ前に安心材料を置く |

### ページ別離脱対策
- **トップページ:** 実績とサービスへの導線を早めに置く
- **サービス詳細:** 関連実績と料金導線を配置する
- **料金ページ:** 小規模相談可能性を明示する
- **実績詳細:** 問い合わせの文脈を自然に入れる
- **お問い合わせ:** 入力項目を絞る

## スマホUX設計

### 基本方針
- スマホを主要利用環境として扱う
- PCの縮小版ではなく、スマホ独自の読みやすさを優先する

### 重要要件
- ファーストビューで情報を詰め込みすぎない
- カード内テキスト量を抑える
- CTAボタンの高さを十分に取る
- フォーム入力をしやすくする
- ギャラリー画像が小さくなりすぎない
- PDFタイトルが読める

### スマホで特に注意する点
- ヘッダーの情報過多
- カードの高さばらつき
- 行間不足
- タップ領域不足
- フォーム項目の多さ
- モーダルの閉じづらさ

## 信頼形成UX設計

本サイトは個人事業主フェーズのため、会社概要の厚さではなく、以下で信頼を作る。

### 信頼形成要素
- 整理されたUI
- 具体的なサービス説明
- 実績の質
- 案件詳細の中身
- 料金の誠実な説明
- 相談しやすい文言
- 公開後の進め方の見える化

### 信頼形成を弱める要因
- 抽象コピーだけ
- 安売り訴求
- 情報の乱雑さ
- 実績の薄さ
- フォームの怖さ
- 長すぎる自己紹介

## マイクロUX設計

### ボタン
- 押せることが一目で分かる / 主ボタンと副ボタンの差が明確 / ホバーや押下変化は軽く上品

### カード
- クリック対象と分かる / 一覧では情報を抑える / 詳細に進みたくなる

### フォーム
- ラベル常時表示 / 必須が分かる / エラーが優しい / 送信後の安心感がある

### ギャラリー
- 画像とPDFが混在しない / 拡大しやすい / 閉じやすい

---

## 不安解消UX設計

### 主要不安と解消手段

| 主要不安 | 解消手段 |
|:---|:---|
| まだ要件が固まっていない | お問い合わせページ歓迎文 / サービス詳細の課題明示 |
| 小規模でもよいのか | 料金ページFAQ的説明 / トップページ不安解消セクション |
| 料金感が分からない | 料金ページの考え方説明 |
| 制作だけでなく開発もできるのか | サービス詳細ページでの対応範囲明示 |
| 何を書いて問い合わせればいいか分からない | お問い合わせページの歓迎文 / 入力項目を絞る |

### 推奨文脈（例）
- 「要件整理からご相談いただけます」
- 「小さく始めて段階的に広げることも可能です」
- 「何を頼めばよいか整理できていない段階でも問題ありません」

## UX改善サイクル

1. 分析データ確認
2. 離脱箇所の特定
3. 仮説作成（例：トップのコピーが抽象的 / 料金ページが弱い / 問い合わせ導線が強すぎるまたは弱すぎる）
4. 改善案設計
5. 文言、導線、UIを修正
6. 再計測

この改善サイクルは運用設計、分析設計と連携する。
